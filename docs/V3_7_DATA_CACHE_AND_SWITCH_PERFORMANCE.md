# V3.7 数据缓存与自选股切换性能修正

## 背景

当前页面出现两个体验问题：

- 数据看起来混乱，部分缓存记录曾被旧区间数据覆盖。
- 切换自选股反应迟钝，尤其在在线数据源慢或不可用时更明显。

## 缓存说明

本项目不是把所有行情数据都直接写进 SQLite：

- SQLite 保存：
  - 自选股。
  - 股票基础信息。
  - 缓存索引，例如缓存 key、数据类型、股票代码、文件路径、数据源、更新时间、TTL。
- Parquet 文件保存：
  - K 线。
  - 实时行情缓存。
  - 分钟线。
  - 资金流等时序数据。

在线读取成功后会写入本地缓存。后续非强制刷新请求优先读本地缓存。

## 问题原因

### 数据混乱

历史上缓存文件路径只按股票代码保存，容易出现不同时间区间的数据互相覆盖。之后虽然增加了缓存 key 哈希文件名，但还缺少“写入前校验”。如果某个数据源返回了明显过旧的日线数据，也可能被写入缓存。

### 切换迟钝

切换自选股时前端会触发 K 线详情请求。此前选中股票变化还会额外触发整组自选股实时行情刷新。如果实时行情源或 K 线源卡住，页面会显得反应慢。

另外本地曾残留多个后端开发进程，其中 `127.0.0.1:8008` 命中了旧进程，而局域网 IP `192.168.22.22:8008` 命中新进程，导致同一个接口在不同地址下速度差异很大。

## 修正内容

- 后端读取缓存时支持“本地优先”：
  - 非强制刷新优先返回已有缓存。
  - 手动刷新才强制访问在线数据源。
- 后端写入缓存前增加数据校验：
  - 默认日线必须通过最新日期校验。
  - 明显过旧的数据不会再写入缓存。
- 前端切换自选股时不再重复刷新整组自选股行情。
- 前端 K 线和行情请求增加请求序号，避免慢响应覆盖新选择的股票。
- Vite 代理目标临时切到 `http://192.168.22.22:8008`，绕开残留的旧 `127.0.0.1` 后端监听。

## 验收结果

- 自选股报价经 Vite 代理返回约 125ms。
- 默认自选股日 K 返回约 35ms 到 321ms。
- 当前默认自选股 K 线最新日期均为 `2026-06-11`。
- 页面显示 `002138 · 使用本地缓存`。
- 浏览器控制台无错误。
