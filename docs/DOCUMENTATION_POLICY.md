# 文档落地规则

## 目的

本项目后续会持续迭代。为了避免方案、升级计划和重要决策只停留在聊天记录里，所有“方案类型”和“更新类型”的内容都必须落地成文档。

## 需要落文档的内容

### 方案类型

包括但不限于：

- 新版本升级方案。
- 架构方案。
- 数据源方案。
- 缓存方案。
- 前端交互方案。
- 指标和分析能力方案。
- 测试和验收方案。

推荐位置：

- 大版本方案：`docs/V*_UPGRADE_PLAN.md`
- 架构方案：`docs/ARCHITECTURE.md`
- 数据方案：`docs/DATA_AND_CACHE.md`
- API 方案：`docs/API_CONTRACT.md`
- 验收方案：`docs/*_ACCEPTANCE.md`

### 更新类型

包括但不限于：

- 已完成的阶段更新。
- 重要 bug 修复。
- 接口变更。
- 数据源切换。
- 缓存策略调整。
- 前端体验调整。
- 验证结果和已知问题。

推荐位置：

- 统一记录：`docs/UPDATES.md`
- 大版本完成后，可同步更新对应升级方案的状态。

## 文档格式

每次更新建议包含：

- 日期。
- 类型：方案更新 / 实现更新 / 修复更新 / 验证更新。
- 背景。
- 改动内容。
- 验证方式。
- 遗留问题。
- 下一步计划。

示例：

```text
## 2026-06-11：V2 图表增强

类型：实现更新

完成内容：
- 增加成交量副图。
- 增加 MACD / RSI tab。

验证：
- npm run test 通过。
- npm --prefix frontend run build 通过。
- 浏览器验证周期切换正常。

遗留问题：
- 资金流字段仍需适配更多数据源差异。
```

## 执行规则

- 用户提出方案、升级、规划、阶段总结时，先创建或更新文档。
- 用户要求“开工”后，再按文档实施。
- 实施完成后，必须更新 `docs/UPDATES.md`。
- 如果方案发生变化，必须同步更新对应方案文档。
